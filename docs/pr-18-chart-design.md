# PR 18 설계 — DB/DC 민감도 차트

> 상태: 설계 확정. 이 PR은 문서와 디자인 계약만 변경하며, 차트 구현·계산식·의존성은 변경하지 않는다.
>
> 구현 상태: 후속 구현 PR에서 네이티브 SVG 차트, 순수 데이터 변환, 단위·E2E 테스트를 완료했다. 계산 엔진과 의존성은 변경하지 않았다.

## 1. 목표

현재 임금상승률을 유지했을 때 DC 운용수익률 변화에 따라 DB와 DC 예상 퇴직급여가 어떻게 달라지는지 한눈에 보여준다.

- 기존 민감도 계산 결과를 시각화한다.
- DB와 DC의 관계 및 손익분기 지점을 빠르게 파악하게 한다.
- 정확한 금액은 기존 표에서 계속 확인할 수 있게 한다.
- 시뮬레이션을 수익 예측이나 투자 권유처럼 표현하지 않는다.

## 2. 범위

### 포함

- 첫 차트의 정보 구조와 문구
- 데이터 입력·표시 계약
- 색상 외 구분, 스크린 리더, 모바일, 인쇄 계약
- 구현 파일 경계와 테스트 기준
- 의존성 도입 판단

### 제외

- React/SVG 컴포넌트 구현
- 계산 엔진 및 기본 민감도 구간 변경
- 툴팁, 확대·축소, 시리즈 토글, 애니메이션
- 차트 라이브러리 추가
- 몬테카를로·스트레스 테스트·세금 결과의 추가 차트
- 서버 저장, 분석 이벤트, 외부 금융 API

## 3. 첫 차트 결정

### DB/DC 예상 퇴직급여 비교선

- 위치: `SensitivitySection`의 **수익률별 결과 (현재 임금상승률 기준)** 제목 아래, 기존 `ReturnRateTable` 위
- 가로축: DC 연평균 운용수익률, `0%`부터 `8%`까지 1%p 간격
- 세로축: 예상 퇴직급여, 원 단위 계산값을 억원 또는 만원 단위의 짧은 눈금으로 표시
- DB 계열: 각 수익률에서 `dbExpectedAmount`
- DC 계열: 각 수익률에서 `dcExpectedAmount`
- 손익분기 표시: `SimulationResult.breakevenReturnRate`가 0~8% 안에 있을 때만 세로 기준선과 수치 라벨 표시
- 현재 입력 표시: `input.dcReturnRate`가 0~8% 안에 있을 때만 점 표식과 `현재 입력` 라벨 표시

DB 선이 수평처럼 보이더라도 계산 결과의 각 점을 사용한다. UI에서 DB/DC 산식을 다시 구현하거나 보간값을 계산하지 않는다.

### 이 차트를 먼저 선택한 이유

- 기존 `ReturnRateTable`과 같은 9개 점을 사용하므로 계산 정합성을 보존한다.
- 두 예상액의 교차가 손익분기 개념과 직접 연결된다.
- 차이 금액만 그리는 방식보다 DB와 DC의 절대 규모를 함께 확인할 수 있다.
- 원형·게이지·3D 차트보다 정확한 비교와 보수적인 제품 인상에 맞는다.

## 4. 데이터 계약

차트는 계산 엔진의 결과만 소비한다.

```ts
interface RetirementComparisonChartDatum {
  dcReturnRate: number;
  dbExpectedAmount: number;
  dcExpectedAmount: number;
}
```

- 입력 원천: `buildSensitivityMatrix(input, [input.wageGrowthRate])`
- 점 순서: `dcReturnRate` 오름차순
- 기본 점 개수: `DEFAULT_RETURN_RATES`와 동일한 9개
- 모든 금액: KRW `number`
- 모든 비율: decimal rate
- 차트용 변환은 순수 TypeScript 함수로 두며 React 컴포넌트와 분리한다.
- 변환 함수는 정렬과 필드 선택만 담당하며 DB/DC 금액, 손익분기, 반올림을 재계산하지 않는다.
- 비정상적으로 비어 있는 데이터는 빈 축을 그리지 않고 한국어 대체 문구와 표를 표시한다.

권장 구현 경계:

```text
src/features/simulator/
  components/RetirementComparisonChart.tsx
  components/RetirementComparisonChart.test.tsx
  utils/chartData.ts
  utils/chartData.test.ts
```

## 5. 시각 계약

- DB: `amber-700` 계열, 점선, 원형 점 표식
- DC: `blue-700` 계열, 실선, 사각형 점 표식
- 선 두께: 기본 2px, 현재 입력 표식은 최소 3px 외곽선
- 격자: 값 판독에 필요한 수평선만 `gray-200`으로 절제해 사용
- 계열 이름은 범례뿐 아니라 선 끝에도 직접 표시한다.
- 손익분기선은 `gray-700` 점선과 `손익분기 약 3.4%` 형식의 텍스트를 함께 사용한다.
- 현재 입력은 `현재 입력 4%`처럼 수치를 포함한다.
- 차트 제목이나 라벨에 `예상 수익`, `권장`, `안전`, `보장` 같은 표현을 사용하지 않는다.
- 모든 금액 표시는 기존 formatter와 같은 반올림 규칙을 사용한다.

화면에 표시할 설명:

> 입력하신 조건과 현재 임금상승률을 기준으로 한 세전 시뮬레이션입니다. 선은 확정 수익이나 운용성과를 보장하지 않습니다.

## 6. 손익분기·범위 밖 처리

- 손익분기 수익률이 0~8% 안이면 차트에 기준선을 표시한다.
- 0% 미만이면 `손익분기 수익률이 표시 범위(0~8%)보다 낮습니다.`라고 요약한다.
- 8% 초과이면 `손익분기 수익률이 표시 범위(0~8%)보다 높습니다.`라고 요약한다.
- `null`이면 `입력하신 조건에서는 손익분기 수익률을 계산할 수 없습니다.`라고 표시한다.
- 현재 입력 수익률이 범위 밖이면 차트 경계로 끌어오지 않고 `현재 입력 수익률은 표시 범위 밖입니다.`라고 표시한다.
- 범위 밖 값 때문에 가로축을 자동 확장하지 않는다. 서로 다른 입력 결과의 시각적 기준을 동일하게 유지하기 위해서다.

## 7. 접근성 계약

- 차트는 `<figure>`와 `<figcaption>`으로 묶는다.
- SVG는 `role="img"`와 연결된 제목·설명을 가진다.
- 스크린 리더 설명에는 축 범위, DB/DC 변화 방향, 손익분기 상태, 현재 입력 상태를 포함한다.
- 계열은 색상만으로 구분하지 않고 선 모양, 점 모양, 직접 라벨을 함께 사용한다.
- 정적 차트에는 포커스 가능한 개별 점이나 hover 전용 툴팁을 만들지 않는다.
- 정확한 9개 값은 기존 `ReturnRateTable`을 항상 DOM에 유지해 대체 데이터 표로 제공한다.
- 차트가 렌더링되지 않아도 표와 텍스트 요약만으로 같은 판단 정보를 얻을 수 있어야 한다.
- 장식용 격자와 축선은 보조 기술에서 숨긴다.

초기 구현은 정적 차트다. 이후 상호작용을 추가한다면 모든 점의 키보드 탐색, 포커스 순서, 툴팁 고정, Escape 닫기까지 별도 설계한 뒤 도입한다.

## 8. 반응형·인쇄 계약

### 모바일

- 고정 픽셀 너비 대신 반응형 SVG `viewBox`를 사용한다.
- 375px에서도 페이지 가로 스크롤이 생기지 않아야 한다.
- 모바일 가로축 라벨은 `0, 2, 4, 6, 8%`로 줄일 수 있지만 9개 데이터 점은 유지한다.
- 세로축 눈금은 최대 4개, 계열 라벨은 겹치면 범례 영역으로 이동한다.
- 최소 차트 높이는 240px로 하되 큰 텍스트 확대에서도 잘리지 않아야 한다.
- 기존 표의 모바일 가로 스크롤 동작은 유지한다.

### 인쇄

- 차트와 표를 모두 인쇄 결과에 포함한다.
- 흑백에서도 실선/점선, 원/사각형 표식, 직접 라벨로 구분되어야 한다.
- 차트 내부에서 페이지가 나뉘지 않도록 `break-inside-avoid`를 적용한다.
- 배경색 인쇄에 의존하지 않는다.

## 9. 구현 원칙

- 초기 구현은 React와 네이티브 SVG로 작성한다.
- 9개 점, 2개 계열, 정적 표시에는 차트 라이브러리를 추가하지 않는다.
- 축 범위·좌표 변환·라벨 간격은 순수 함수로 테스트할 수 있게 분리한다.
- `src/calculator/**`는 변경하지 않는다.
- 차트와 표는 동일한 `currentGrowthMatrix` 인스턴스를 받는다.
- 입력 변경 시 애니메이션 없이 즉시 갱신한다.
- 정적 SVG에 HTML 문자열 삽입이나 사용자 제공 텍스트를 사용하지 않는다.

차트 요구가 다중 축, 확대·축소, 수백 개 점으로 커진다면 별도 PR에서 접근성·번들 크기·SSR 호환성을 비교한 뒤 의존성을 재검토한다.

## 10. 테스트 계약

### 단위 테스트

- 9개 민감도 점이 오름차순 차트 데이터로 변환됨
- 원 단위 금액과 decimal rate가 변형되지 않음
- 빈 데이터와 동일값 세로축에서 유한한 좌표를 반환함
- 손익분기 및 현재 입력의 범위 안/아래/위/null 상태 판정
- 모바일 축 라벨 축약이 데이터 점을 삭제하지 않음

### 컴포넌트 테스트

- 한국어 제목, 설명, 두 계열 직접 라벨 렌더링
- SVG의 접근 가능한 이름과 설명 연결
- DB/DC가 선·점 모양으로도 구분됨
- 범위 밖 및 계산 불가 문구 렌더링
- 기존 표가 함께 유지됨

### Playwright

- 기본 입력에서 차트·표·손익분기 라벨 표시
- DB 우세, DC 우세, 손익분기 범위 밖 시나리오
- 375px, 768px, 1280px에서 가로 오버플로·라벨 잘림 없음
- 인쇄 미디어에서 차트와 표 노출
- axe 위반, console error, page error 없음

전체 회귀 기준은 구현 시점의 `pnpm qa`이며 계산 golden 값은 기존 테스트를 그대로 통과해야 한다.

## 11. 완료 기준과 후속
